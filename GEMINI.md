## 🏛️ Architecture Conventions

### 1. Window Reference Safety (`.ptyWindow()`)
* **Standard:** When referencing an active `PseudoTerminal` window's instance, **never** call `.window()` in Swift files.
* **Why:** `PseudoTerminal` inherits a `.window` property from `NSWindowController` but also conforms to `iTermWindowController` which defines a `.window()` method. This selector clash triggers dynamic dispatch failures and spontaneous crashes.
* **Implementation:** Always call the dedicated, non-clashing C-level method `.ptyWindow()`.
  ```swift
  // Correct
  guard let wn = terminal.ptyWindow()?.windowNumber else { return }
  ```

### 2. Nullability of terminals (`iTermController.h`)
* **Standard:** The `terminals` property on `iTermController` must remain explicitly marked as `nullable` in Objective-C:
  ```objc
  - (nullable NSArray<PseudoTerminal *> *)terminals;
  ```
* **Why:** During application shutdown and teardown, `_terminalWindows` is set to `nil`. Marking it `nullable` ensures that the Swift runtime bridges it to `[PseudoTerminal]?`, preventing runtime type-casting assertion crashes at exit.

### 3. Action Button Target Binding
* **Standard:** When configuring outline view buttons or toolbar panel buttons, **always** explicitly pass `self` (the controller instance) as the target of the button, rather than setting `target: nil`.
* **Why:** Setting `target: nil` relies on Cocoa's responder chain, which gets broken/hijacked when keyboard or selection focus changes, disabling the buttons. Explicit target binding keeps buttons permanently active.

### 4. Process Survival & Re-attachment
* **Standard:** To enable active shell processes (`iTerm2Server`) to successfully re-attach when restored from cold storage:
  * Capture active session dimensions (`firstSession.columns` and `firstSession.rows`) and inject them under the `"Archive"` key in the arrangement.
  * Clear `_isArchive` in `PTYSession.m` inside the `if (didAttach)` block so that the restored terminal is not marked as a dead read-only archive, leaving it fully active and interactive.

## 🧪 Testing Guidelines

### 1. Unit Tests (`ModernTests/iTermWindowProjectsTests.swift`)
* **Scope:** Covers tree hierarchies, nested project CRUD, JSON persistence schema integrity, outline view data-source mappings, and button state machines.
* **TDD Principle:** Write failing unit tests first. Swizzle environment settings (like `runJobsInServers`) or attachment outcomes dynamically using a `defer` unswizzle block to isolate state changes.
* **Target Safety:** Keep all graphical window creation out of the headless target (`ModernTests`) as Cocoa sandboxing blocks real window-server calls under `xcodebuild`.

### 2. E2E System Integration Tests (`tools/test_cold_storage_e2e.sh`)
* **Scope:** Full OS-level verification of process survival, freeze/thaw, and terminal unfreezing.
* **Isolation Standard:** To avoid interfering with a user's running **production iTerm instance**:
  * Launch the development build via its absolute path.
  * Query its specific Process ID (`DEV_PID = $(pgrep -f "$APP_PATH/Contents/MacOS/iTerm2")`).
  * Wrap all UI scripting in `tell application "System Events" to tell process id DEV_PID`.
